## 1. Executive Summary

v1.0.2 is the first content-and-polish update following the v1.0.1 hotfix. It focuses on four pillars: (1) visual juice through character animation, (2) deeper meta-progression hooks, (3) competitive differentiation features, and (4) performance hardening for sustained play sessions. The roadmap is organized into three tiers by impact and risk.

## 3. Feature Assessment Matrix

Each candidate feature is scored on four dimensions (1-5 scale):

- **Player Impact**: How much does this improve retention, satisfaction, or "one more run" motivation?
- **Implementation Cost**: 5 = low cost (< 100 lines, no new systems), 1 = high cost (> 500 lines, new systems)
- **Regression Risk**: High/Medium/Low chance of breaking existing functionality
- **Differentiation Value**: How unique is this feature compared to VS/Brotato/HoloCure?

## 4. Feature Candidates

### 4.1 Character Animation System (Sprite2D + _physics_process)

**Priority**: P1 HIGH
**Version Target**: v1.0.2

| Dimension | Score | Rationale |
|---|---|---|
| Player Impact | 3 | Visual juice. Current static sprites feel dead during movement. Animation creates a sense of life. |
| Implementation Cost | 4 | Art spec already complete (`character-animation-integration.md`). Method B (Timer) or C (_physics_process) avoids .tscn changes. ~30 lines in player.gd. |
| Regression Risk | Low | Additive visual change. Sprite2D node type preserved. Afterimage system needs one line update. |
| Differentiation | 1 | Standard for all competitors. Not a differentiator, but a baseline expectation. |

**Spec reference**: `docs/superpowers/specs/character-animation-integration.md` (Method C: Sprite2D + _physics_process, selected as best balance of simplicity and compatibility).

**Implementation scope**:
- Add `_anim_accumulator: float` and `_anim_frame: int` variables to player.gd
- In `_physics_process()`, accumulate delta and swap texture at 4 FPS when moving
- Two textures per character: idle (existing .png) + action (existing _cast/_block/_draw.png)
- Fix afterimage: change `sprite.texture` reference to use the current frame texture
- Estimated: ~25 lines added, 1 line modified

**Blocker**: None. Art assets already exist (mage_cast.png, warrior_block.png, ranger_draw.png).

### 4.2 Daily Challenge Mode

**Priority**: P2 MEDIUM
**Version Target**: v1.0.2 or v1.1.0

| Dimension | Score | Rationale |
|---|---|---|
| Player Impact | 5 | Daily challenges are the strongest retention mechanic in the genre. VS daily challenges drive >50% of long-term sessions. |
| Implementation Cost | 2 | Requires: seeded RNG system, daily seed generation (date-based), daily leaderboard (even local), daily reward structure, UI entry point. ~300-400 lines across multiple files. |
| Regression Risk | Medium | New game mode, isolated from main path. Risk is in RNG seeding affecting existing systems. |
| Differentiation | 4 | Brotato has no daily. H5 has no daily. VS daily is the gold standard. Adding this positions the game competitively. |

**Design overview**:
- Daily seed derived from date string (e.g., hash("2026-04-17"))
- Seeded RNG controls: enemy spawn order, upgrade options, weapon availability, boss timing
- All players on the same day get identical conditions
- Local "best" tracking per day (no server required)
- Entry point: button on title screen, only visible after first victory
- Reward: 2x soul fragments for daily runs

**Why defer to v1.0.2 or v1.1.0**: Seeded RNG is a new system that needs careful isolation. The current RNG (randf/randf_range) is scattered across 10+ files. Implementing seeded RNG requires a centralized RandomNumberGenerator instance passed through the game pipeline.

### 4.3 Weapon Mastery System (Kill-Tracking Upgrades)

**Priority**: P2 MEDIUM
**Version Target**: v1.0.2

| Dimension | Score | Rationale |
|---|---|---|
| Player Impact | 4 | Gives long-term progression beyond shop upgrades. "I want to master the boomerang" is a powerful motivator for repeated runs. |
| Implementation Cost | 2 | New UI (mastery panel), kill tracking per weapon, mastery thresholds (e.g., 50/200/500 kills), mastery rewards (titles, visual flair). ~200-300 lines. |
| Regression Risk | Low | Additive tracking system. Kill attribution already implemented via weapon_id. |
| Differentiation | 3 | H5 has no weapon mastery. VS has implicit weapon familiarity through achievements. Brotato tracks character wins. |

**Design overview**:
- Track kills per weapon type in SaveManager
- Mastery tiers: Novice (0), Adept (50), Expert (200), Master (500)
- Rewards per tier: visual badge on weapon slot, title suffix, +2%/+4%/+6% damage with that weapon type
- Mastery progress visible in shop screen
- Creates long-term grind incentive: "Master all 7 weapons" achievement

**Key design decision**: Weapon mastery damage bonus is multiplicative with shop weaponDmg upgrade. A Master weapon with Lv3 shop upgrade has 1.10 * 1.06 = 1.166x damage. This is a meaningful but not game-breaking bonus.

### 4.4 Audio System (BGM + SFX)

**Priority**: P1 HIGH
**Version Target**: v1.0.2

| Dimension | Score | Rationale |
|---|---|---|
| Player Impact | 5 | A silent game loses 40% of its emotional impact. Audio is the single highest-value missing feature. |
| Implementation Cost | 2 | Requires audio assets (compose/procure BGM + SFX), AudioStreamPlayer nodes, audio bus configuration, crossfade system, SFX trigger points across 15+ scripts. ~400 lines + asset procurement. |
| Regression Risk | Low | Additive system. Audio failures are non-blocking (game works without sound). |
| Differentiation | 1 | Baseline expectation, not a differentiator. |

**Implementation approach**:
- Godot AudioStreamPlayer for BGM (loop), AudioStreamPlayer2D for positional SFX
- BGM: 1 looping track per game phase (menu, arena, boss, victory, game over)
- SFX categories: weapon fire (7), hit/enemy death, player damage, level up, coin pickup, combo milestone, dash, skill activation
- Volume bus hierarchy: Master > BGM (0.7) / SFX (1.0) / UI (0.8)
- Audio toggle in title screen settings

**Blocker**: Asset procurement. Need BGM composition or royalty-free sourcing. SFX can use Godot built-in generators for prototype.

### 4.7 Performance: Object Pool System

**Priority**: P2 MEDIUM
**Version Target**: v1.0.2

| Dimension | Score | Rationale |
|---|---|---|
| Player Impact | 2 | Invisible to most players. Only noticeable in endless mode with 100 enemies + many projectiles. |
| Implementation Cost | 2 | Generic ObjectPool<T> system, pre-warm pools for projectiles/enemies/xp_gems. ~200 lines new code, 15+ file modifications. |
| Regression Risk | Medium | Changes entity lifecycle from create/destroy to acquire/release. If release() misses cleanup, state leaks between entities. |
| Differentiation | 0 | Internal optimization, invisible to players. |

**Design**:
- Pre-warm pools: 100 projectiles, 70 enemies, 200 xp_gems
- Pool nodes stored in ProjectileManager (existing)
- entity.queue_free() replaced by entity.return_to_pool()
- Pool returns node to inactive state (reset position, velocity, visibility)

**Why v1.0.2**: v1.0.1's enemy cache already reduces the hottest bottleneck (get_nodes_in_group). Object pool targets the second bottleneck (instantiate/queue_free churn during intense combat). Better to stabilize the cache change first.

## 7. Decision Records

| Decision | Why | Alternative Considered |
|---|---|---|
| Character Animation is v1.0.2 #1 | Lowest cost with visible improvement. Art assets exist. Spec complete. No blockers. | Defer to v1.1.0 (delays visual polish too long) |
| Audio System is v1.0.2 #2 | Highest player impact (5/5). Silent gameplay is the biggest experiential gap. Assets are the only blocker. | Generate placeholder audio in code (insufficient quality), defer to v1.1.0 (too long without audio) |
| Weapon Mastery in v1.0.2 | 4/5 player impact with 2/5 cost. Kill tracking already implemented. Natural extension of existing meta-progression. | Defer to v1.1.0 (delays long-term retention hook) |
| Daily Challenge deferred to v1.1.0 | 5/5 impact but requires seeded RNG system (affects 10+ files). Too risky for a polish update. | Implement without seeded RNG (loses the "same conditions" competitive element) |
| Object Pool is stretch goal | v1.0.1 enemy cache already targets the hottest bottleneck. Pool targets secondary bottleneck. Stabilize cache first. | Implement in v1.0.2 core (adds risk, cache + pool simultaneous is too much perf work at once) |
| 3 unregistered evolved weapons are stretch | These are Godot-original content (P2). 9 evolved weapons already exceed H5's 8. Not blocking any player-facing issue. | Register immediately (scope creep, no player demand yet) |

## 8. Success Criteria

v1.0.2 is considered complete when:

1. All 1276+ existing tests still pass (zero regressions)
2. Characters animate when moving (2-frame walk cycle at 4 FPS)
3. Audio plays during gameplay (BGM + core SFX categories)
4. Weapon mastery progress visible in shop screen
5. Tutorial covers synergies and evolution hints (steps 6-8)
6. Project score >= 95/100
7. Total test count >= 1350
8. No new P0 or P1 bugs introduced
